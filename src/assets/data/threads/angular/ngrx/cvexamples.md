## 1. The Facade Pattern — Decoupling Components from the Store

### Why Use a Facade?
At enterprise scale, components must not couple directly to NgRx. A Facade service acts as the boundary — components call the facade, the facade dispatches actions and exposes selectors as observables.

**CV hook:** *"At Aumovio I led Angular frontend architecture using component factories, singletons, stores and services — the Facade pattern kept feature modules decoupled and enabled 30% faster client deliverables."*

```typescript
// survey.facade.ts
@Injectable({ providedIn: 'root' })
export class SurveyFacade {
  surveys$ = this.store.select(selectAllSurveys);
  loading$  = this.store.select(selectSurveysLoading);
  kpis$     = this.store.select(selectFilteredSurveyKPIs);

  constructor(private store: Store) {}

  loadSurveys() {
    this.store.dispatch(SurveyActions.loadSurveys());
  }

  createSurvey(survey: Survey) {
    this.store.dispatch(SurveyActions.createSurvey({ survey }));
  }

  applyFilter(filter: SurveyFilter) {
    this.store.dispatch(SurveyActions.setFilter({ filter }));
  }
}
```

```typescript
// survey-list.component.ts — zero NgRx imports
@Component({ /* ... */ })
export class SurveyListComponent {
  surveys$ = this.facade.surveys$;
  loading$ = this.facade.loading$;

  constructor(private facade: SurveyFacade) {}

  ngOnInit() { this.facade.loadSurveys(); }
}
```

**Benefits:**
- Components have zero imports from `@ngrx/store`
- Swapping state management only touches the facade
- Facades are trivial to mock in tests (one spy object)

## 2. Feature State with Lazy Loading

### Pattern
Register state only when the feature module loads. Critical in large apps (Survey Platform, AI Reviewer) where startup performance matters.

```typescript
// survey.module.ts
@NgModule({
  imports: [
    StoreModule.forFeature('surveys', surveyReducer),
    EffectsModule.forFeature([SurveyEffects]),
  ]
})
export class SurveyModule {}
```

```typescript
// app-routing.module.ts
const routes: Routes = [
  {
    path: 'surveys',
    loadChildren: () =>
      import('./features/survey/survey.module').then(m => m.SurveyModule)
  }
];
```

**Interview talking point:** *"In the Survey Management Platform we used lazy-loaded feature modules. Each feature registered its own slice of NgRx state via `StoreModule.forFeature`, so the global store only carried the state of what was actually loaded — this kept startup bundle small."*

## 3. Effects — Choosing the Right Flattening Operator

```typescript
@Injectable()
export class SurveyEffects {
  // switchMap: cancel in-flight load on re-trigger (navigation change)
  loadSurveys$ = createEffect(() =>
    this.actions$.pipe(
      ofType(SurveyActions.loadSurveys),
      switchMap(() =>
        this.surveyService.getAll().pipe(
          map(surveys => SurveyActions.loadSurveysSuccess({ surveys })),
          catchError(error => of(SurveyActions.loadSurveysFailure({ error: error.message })))
        )
      )
    )
  );

  // concatMap: queue writes in order (sequential survey submissions)
  createSurvey$ = createEffect(() =>
    this.actions$.pipe(
      ofType(SurveyActions.createSurvey),
      concatMap(({ survey }) =>
        this.surveyService.create(survey).pipe(
          map(created => SurveyActions.createSurveySuccess({ survey: created })),
          catchError(error => of(SurveyActions.createSurveyFailure({ error: error.message })))
        )
      )
    )
  );

  constructor(private actions$: Actions, private surveyService: SurveyService) {}
}
```

### Operator Decision Table

| Operator | Behaviour | Use Case |
|---|---|---|
| `switchMap` | Cancels previous | Load on route change, search autocomplete |
| `mergeMap` | All concurrent | Independent parallel requests |
| `concatMap` | Ordered queue | Sequential form submissions, ordered writes |
| `exhaustMap` | Ignores until done | Login button, prevent double-submit |

**Interview talking point:** *"We used `switchMap` for survey list loads because navigating away should cancel the in-flight request. For creating entries we used `concatMap` to guarantee ordered writes to the backend."*

## 4. Selector Composition for KPI Dashboards

The Survey Platform required custom KPI tracking and granular data filtering. Selector composition was the mechanism.

```typescript
// Feature selector
export const selectSurveyState = createFeatureSelector<SurveyState>('surveys');

// Entity selectors (generated by NgRx Entity adapter)
const { selectAll } = adapter.getSelectors();
export const selectAllSurveys  = createSelector(selectSurveyState, selectAll);
export const selectActiveFilter = createSelector(selectSurveyState, s => s.activeFilter);
export const selectDateRange    = createSelector(selectSurveyState, s => s.dateRange);

// Composed KPI selector — only recalculates when inputs change (memoization)
export const selectFilteredSurveyKPIs = createSelector(
  selectAllSurveys,
  selectActiveFilter,
  selectDateRange,
  (surveys, filter, dateRange) => {
    const filtered = surveys
      .filter(s => !filter.department || s.department === filter.department)
      .filter(s => s.createdAt >= dateRange.start && s.createdAt <= dateRange.end);

    return {
      total:        filtered.length,
      avgScore:     filtered.reduce((sum, s) => sum + s.score, 0) / (filtered.length || 1),
      responseRate: filtered.filter(s => s.completed).length / (filtered.length || 1)
    };
  }
);
```

**Interview talking point:** *"The KPI dashboard was entirely derived state from selectors. Because `createSelector` memoizes, the dashboard only recalculated when the underlying filter or survey data actually changed — the component received zero unnecessary updates."*

## 5. Role-Based UI with Store Selectors

The Survey Platform had role-based access control. The store held user roles; selectors derived UI permissions.

```typescript
// auth.selectors.ts
export const selectCurrentUserRoles = createSelector(selectAuthState, s => s.roles);

export const selectCanCreateSurvey = createSelector(
  selectCurrentUserRoles,
  roles => roles.includes('ADMIN') || roles.includes('SURVEY_MANAGER')
);

export const selectCanViewAnalytics = createSelector(
  selectCurrentUserRoles,
  roles => roles.some(r => ['ADMIN', 'ANALYST', 'MANAGER'].includes(r))
);
```

```html
<!-- Component template -->
<button *ngIf="canCreate$ | async" (click)="create()">New Survey</button>
<app-analytics *ngIf="canViewAnalytics$ | async"></app-analytics>
```

**Pattern:** Never put role checks in components directly — always derive from store selectors. Single source of truth for authorization, easy to test, easy to audit.
